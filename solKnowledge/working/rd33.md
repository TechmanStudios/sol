# **CLAIM 1 — Hold duration is a strong regime knob; hold=400 “programs” a distinct bus regime**

**CLAIM**  
 Under these Phase 3.11 conditions, `betweenRepTicks_hold = 400` reliably shifts the system into a different outcome regime vs holds `0` or `81`, including: (i) a large increase in `winner_peakBus = 136` frequency, (ii) reduced 114 dominance fraction, (iii) reduced bus dominance entropy, and (iv) larger peak bus magnitudes.

**EVIDENCE** (MASTER\_summary; combined across BN15+BN16, n=160 reps per hold)

* **136 peak-win probability (`winner_peakBus=136`)**

  * hold=0: `p136 = 0.01875` (3/160)

  * hold=81: `p136 = 0.01875` (3/160)

  * hold=400: `p136 = 0.21250` (34/160)

* **Mean dominance fraction**

  * hold=0: `mean fracTicks_busDom114 = 0.204473`

  * hold=81: `mean fracTicks_busDom114 = 0.201559`

  * hold=400: `mean fracTicks_busDom114 = 0.190446` (systematic drop)

* **Mean dominance entropy**

  * hold=0: `mean busDomEntropy_bits = 0.730077`

  * hold=81: `mean busDomEntropy_bits = 0.723927`

  * hold=400: `mean busDomEntropy_bits = 0.701569` (systematic drop)

* **Mean late capture timing (primary)**

  * hold=0: `mean lateCaptureTick_114_primary = 127.856`

  * hold=81: `mean lateCaptureTick_114_primary = 114.362`

  * hold=400: `mean lateCaptureTick_114_primary = 96.275` (earlier)

* **Mean peak magnitudes**

  * hold=0: `mean peakAbs114_bus = 1.447`, `mean peakAbs136_bus = 1.401`

  * hold=81: `mean peakAbs114_bus = 1.895`, `mean peakAbs136_bus = 1.834`

  * hold=400: `mean peakAbs114_bus = 4.048`, `mean peakAbs136_bus = 3.863` (large increase)

**ASSUMPTIONS**

* `winner_peakBus`, dominance fractions, and entropy are valid macro readouts for “regime”.

* These effects are not artifacts of pass ordering (addressed by Claim 2 \+ counterbalance).

**FALSIFY**

* Change only the hold set (e.g., test `{0, 81, 200, 400, 800}`) and see whether regime shifts scale smoothly with hold or show thresholds.

* Repeat BN15/BN16 pair at different `dampUsed` (e.g., 18, 22\) to see whether the hold=400 regime persists.

**STATUS:** **robust** (replicated across AB then BA counterbalance).

# **CLAIM 2 — Macro outcomes are largely reversible w.r.t. hold-order swap (A vs B), even when pass order is reversed**

**CLAIM**  
 Swapping hold order between passes does **not** materially change macro outcomes for the key regime (hold=400). The most obvious macro signature—136 win rate at hold=400—remains stable across passes in both BN15 and BN16.

**EVIDENCE** (MASTER\_summary; winner counts out of 20 reps per (dir, pass, hold))

* **BN15 hold=400:** identical across passes within each dir

  * down: A `114:15 / 136:5`, B `114:15 / 136:5`

  * up: A `114:16 / 136:4`, B `114:16 / 136:4`

* **BN16 hold=400:** identical across passes within each dir

  * down: A `114:16 / 136:4`, B `114:16 / 136:4`

  * up: A `114:16 / 136:4`, B `114:16 / 136:4`

* **Macro deltas are small elsewhere:** differences in mean `fracTicks_busDom114` between B and A are on the order of `~±0.006` per (dir, hold), not regime-flipping.

**ASSUMPTIONS**

**FALSIFY**

# **CLAIM 3 — The small late-capture timing shifts are primarily pass-position drift, not an order-label effect**

**CLAIM**  
 The subtle shifts in `lateCaptureTick_114_primary` are not stable with respect to the order label (A vs B). When pass order is reversed (BN16), the sign of the “second minus first pass” delta flips for holds 0 and 81, indicating a **pass-position / time-in-session** component rather than a persistent “A vs B” effect.

**EVIDENCE** (MASTER\_summary; mean delta \= passPosition2 − passPosition1, averaged across dirs)

* **BN15 (A first, B second):**

  * hold=0: `+1.300`

  * hold=81: `−1.700`

  * hold=400: `−1.150`

* **BN16 (B first, A second):**

  * hold=0: `−3.275` *(sign flips vs BN15)*

  * hold=81: `+1.550` *(sign flips vs BN15)*

  * hold=400: `−0.700` *(same sign as BN15; smaller magnitude)*

**INTERPRETATION**

* Holds 0 and 81: timing deltas are dominated by pass-position (session evolution), not stable hold-order labeling.

* Hold 400: second-pass earlier capture persists in both runs, suggesting an interaction between “session state” and the long-hold regime.

**ASSUMPTIONS**

* `lateCaptureTick_114_primary` is sensitive to subtle temporal structure and can act as a drift marker.

**FALSIFY**

* Repeat BN16 again (BA/AB) and check whether the pass-position deltas are consistent run-to-run.

* Add a hard reset between dirs (or rerun in a fresh session) to separate “within-session” drift from protocol effects.

**STATUS:** **supported** (counterbalance broke label-stability; pass-position matters).

## **NET TAKEAWAY (FOR “COMPUTING SUBSTRATE” AIMS)**

\[INTERPRETATION\]

* These two runs together support a clean story: **hold duration behaves like an instruction parameter** that changes regime statistics (including energy scale and winner distribution), while **hold-order swap is macro-reversible**.

* The “micro-timing” features (late capture tick) look like a **sensitive drift probe** rather than a stable order-label memory—useful as instrumentation, not yet a reliable control primitive.

\[HYPOTHESIS\]

* The hold=400 regime may be acting like a **state-charging / energy accumulation interval** (consistent with much larger `peakAbs*` magnitudes), which then shifts competition outcomes toward allowing 136 peaks more often.
